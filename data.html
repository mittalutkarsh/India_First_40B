B.1 · Pre-training data — provenance-tiered, native-first

Every Indic document is tagged by origin. Tier definitions and the native-first rule follow directly from the repetition result: unique native content is scarce and precious, so we repeat it rather than dilute with manufactured text.

  • T0 native-verified — human-checked native text: Sangraha verified (curated sites, OCR'd Indic PDFs, transcribed Indic audio/video), native Wikipedia, digitised books.
  • T1 native-web — native-origin web, perplexity-filtered: Sangraha filtered set + IndicCorp v2.
  • T2 native-synthetic — generated bottom-up from Indian sources (in the spirit of AI4Bharat's IndicAlign and Microsoft's Updesh), generated once, mixed with real data. Capped.
  • T3 translated — English→Indic, minimized; only to reach languages whose native corpus is too thin.

Repetition / effective-token accounting (the ≤4-epoch cap is from Muennighoff et al., 2023):

budget_ℓ  =  e_ℓ · U_ℓ  +  G_ℓ     with   e_ℓ ≤ 4

For a language ℓ: U_ℓ = unique native tokens (T0+T1), e_ℓ = epochs of repetition (capped at ~4, where the loss penalty is still <1%), G_ℓ = manufactured residual (T2+T3). This is where fairness enters the data, not just the tokenizer: Hindi has large U_ℓ and needs few epochs and little G_ℓ; Malayalam and Odia have small U_ℓ, hit the epoch cap sooner, and need proportionally more G_ℓ — which is exactly why they need the closest evaluation.

Curriculum

From-scratch training dilutes low-resource languages if everything is mixed uniformly from the start (the ALLaM finding). Phase A (~60% of tokens) is high-resource-heavy; Phase B (~30%) ramps Indic plus the manufactured top-up; Phase C (~10%) anneals on highest-quality native Indic — the translationese gap-closer (Doshi et al., 2024).
